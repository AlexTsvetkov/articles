# The SAP Commerce Type System Deep Dive: From items.xml to Database and Beyond

The type system is the beating heart of SAP Commerce. It defines every data structure in the platform — from products and orders to customers and CMS components. Unlike traditional Java applications where you write JPA entities and manage database migrations with Flyway, SAP Commerce uses a **metadata-driven approach**: you declare your data model in XML, and the platform generates Java classes, database schemas, and runtime type metadata automatically.

This article is the definitive deep dive into that system. Whether you're a developer extending the OOTB model for the first time or a senior architect debugging type system internals, this guide covers everything from the XML syntax to the generated bytecode to the database tables underneath.

---

## Why a Metadata-Driven Type System?

Before diving into mechanics, it's worth understanding *why* SAP Commerce doesn't use JPA/Hibernate like most Spring applications.

### The Problem with Traditional ORM in Enterprise Commerce

In a standard Spring Boot application, you'd define entities like this:

```java
@Entity
@Table(name = "products")
public class Product {
    @Id @GeneratedValue
    private Long id;
    
    @Column(nullable = false)
    private String code;
    
    @Column
    private String name;
    // ...
}
```

This works well for applications where a single team owns the data model. But SAP Commerce is a **platform** — it ships with hundreds of pre-defined types (Product, Category, Order, Cart, Customer, CMS components, etc.), and every customer project needs to extend those types differently. If the model were defined in Java annotations:

- **Extensibility would require inheritance or composition hacks** — you'd need to subclass `Product` as `MyProduct`, update every query, every service, every DAO
- **Multiple extensions couldn't independently add attributes** to the same type — two extensions adding fields to `Product` would conflict
- **Schema migration would be manual** — every project would need custom Flyway scripts that interact with the platform's schema
- **Localization would require manual implementation** — no built-in multi-language support for attributes

### The SAP Commerce Solution

The type system solves all of these problems:

1. **Declarative extension**: Any extension can add attributes to any existing type by declaring them in `items.xml` with `autocreate="false"`
2. **Automatic merging**: The platform merges all `items.xml` files from all extensions at build time into a single unified type system
3. **Generated code**: Java Model classes are generated from the merged definitions — no manual entity coding
4. **Automatic DDL**: Database schema is generated and migrated automatically via `ant updatesystem`
5. **Built-in localization**: Any attribute can be made localizable by simply changing its type to `localized:java.lang.String`
6. **Runtime metadata**: The complete type system is queryable at runtime — the Backoffice UI, for example, generates its CRUD screens entirely from type metadata

## The Type Hierarchy

Every piece of data in SAP Commerce is an **Item**. The type hierarchy is:

```
                        ┌─────────────┐
                        │    Item      │  (abstract root — PK, creation/modification time)
                        └──────┬──────┘
                               │
              ┌────────────────┼────────────────┐
              │                │                │
     ┌────────▼───────┐ ┌─────▼──────┐  ┌──────▼──────┐
     │  GenericItem    │ │  CronJob   │  │  Type       │
     │  (no deployment │ │            │  │  (metadata) │
     │   needed)       │ │            │  │             │
     └────────┬───────┘ └────────────┘  └─────────────┘
              │
    ┌─────────┼──────────┬──────────────┐
    │         │          │              │
┌───▼──┐ ┌───▼───┐ ┌────▼────┐  ┌─────▼──────┐
│Product│ │ Order │ │Customer │  │Your Custom │
│      │ │       │ │(User)   │  │   Type     │
└──────┘ └───────┘ └─────────┘  └────────────┘
```

### Core Meta-Types

The type system itself is described by meta-types — types that describe other types:

| Meta-Type | Description | Examples |
|-----------|-------------|----------|
| **ComposedType** | Describes an item type (like a Java class definition) | `Product`, `Order`, `Customer` |
| **AtomicType** | Wraps a Java primitive or simple type | `java.lang.String`, `java.lang.Integer`, `java.util.Date` |
| **CollectionType** | A typed collection of another type | `StringCollection` (List of Strings) |
| **MapType** | A typed key-value map | `localized:java.lang.String` (language → string map) |
| **EnumType** | An enumeration with fixed or dynamic values | `ArticleApprovalStatus`, `OrderStatus` |
| **RelationType** | Describes a relationship between two ComposedTypes | `Product2CategoryRelation` |

You can think of `ComposedType` as analogous to `java.lang.Class` — it's the runtime descriptor for an item type. Every `ComposedType` has:
- A **code** (e.g., `"Product"`)
- A **supertype** (parent `ComposedType`)
- A set of **AttributeDescriptor** instances (the type's attributes)
- A **deployment** (database table mapping)
- A **jaloclass** (legacy Jalo layer class)

## items.xml Anatomy: The Complete Reference

The `items.xml` file is the primary artifact for defining the data model. Let's examine every element.

### Root Structure

```xml
<?xml version="1.0" encoding="ISO-8859-1"?>
<items xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
       xsi:noNamespaceSchemaLocation="items.xsd">

    <atomictypes>
        <!-- Rarely used — define new atomic types wrapping Java classes -->
    </atomictypes>

    <collectiontypes>
        <!-- Define typed collections -->
    </collectiontypes>

    <enumtypes>
        <!-- Define enumerations -->
    </enumtypes>

    <maptypes>
        <!-- Define typed maps -->
    </maptypes>

    <relations>
        <!-- Define relationships between item types -->
    </relations>

    <itemtypes>
        <!-- Define item types (the main section) -->
    </itemtypes>

</items>
```

### `<atomictypes>`

Atomic types wrap Java classes. The platform pre-defines most common ones (`java.lang.String`, `java.lang.Integer`, `java.lang.Boolean`, `java.util.Date`, etc.). You rarely need to add new ones unless integrating a custom Java type:

```xml
<atomictypes>
    <atomictype class="com.mycompany.types.Money" autocreate="true" generate="false"/>
</atomictypes>
```

### `<collectiontypes>`

Collection types define typed lists or sets:

```xml
<collectiontypes>
    <collectiontype code="StringCollection" elementtype="java.lang.String" type="list"/>
    <collectiontype code="ProductList" elementtype="Product" type="list"/>
    <collectiontype code="CategorySet" elementtype="Category" type="set"/>
</collectiontypes>
```

The `type` attribute can be `list`, `set`, or `collection`. Under the hood, collection-type attributes are serialized to a single database column (a comma-separated PK list or serialized blob), which has **significant performance implications** — more on this later.

### `<enumtypes>`

Enumerations can be **static** (fixed at build time) or **dynamic** (extensible at runtime):

```xml
<enumtypes>
    <!-- Static enum — values are fixed, generates a Java enum -->
    <enumtype code="OrderStatus" autocreate="true" generate="true" dynamic="false">
        <value code="CREATED"/>
        <value code="CONFIRMED"/>
        <value code="SHIPPED"/>
        <value code="DELIVERED"/>
        <value code="CANCELLED"/>
    </enumtype>

    <!-- Dynamic enum — values can be added at runtime via ImpEx -->
    <enumtype code="LoyaltyTier" autocreate="true" generate="true" dynamic="true">
        <value code="BRONZE"/>
        <value code="SILVER"/>
        <value code="GOLD"/>
        <value code="PLATINUM"/>
    </enumtype>
</enumtypes>
```

**Static enums** (`dynamic="false"`):
- Generate a Java `enum` class
- Values are compile-time constants
- Cannot be extended without rebuilding
- Better type safety in Java code

**Dynamic enums** (`dynamic="true"`):
- Generate a Java class extending `HybrisEnumValue`
- Values are stored in the database (`enumerationvalues` table)
- New values can be added via ImpEx at runtime without a build
- Used when business users need to add values (e.g., new loyalty tiers)

```java
// Static enum usage
if (order.getStatus() == OrderStatus.SHIPPED) { ... }

// Dynamic enum usage  
LoyaltyTier tier = LoyaltyTier.valueOf("GOLD");
if (tier.equals(customer.getLoyaltyTier())) { ... }
```

### `<maptypes>`

Map types define typed key-value pairs:

```xml
<maptypes>
    <maptype code="String2StringMap" 
             argumenttype="java.lang.String" 
             returntype="java.lang.String"/>
    <maptype code="localized:java.lang.String" 
             argumenttype="Language" 
             returntype="java.lang.String"/>
</maptypes>
```

The `localized:` prefix is a special convention — `localized:java.lang.String` is actually a map from `Language` to `String`, enabling built-in multi-language support. When you define an attribute of type `localized:java.lang.String`, the platform automatically:
- Creates a column in the `*lp` (localized properties) table
- Resolves the value based on the current session language
- Supports querying with `{name[en]}` syntax in FlexibleSearch

### `<relations>`

Relations define relationships between item types. This is one of the most important sections:

```xml
<relations>
    <!-- One-to-Many: One Customer has many Orders -->
    <relation code="Customer2OrderRelation" localized="false">
        <sourceElement type="Customer" qualifier="customer" cardinality="one">
            <modifiers optional="false"/>
        </sourceElement>
        <targetElement type="Order" qualifier="orders" cardinality="many"
                       collectiontype="set" ordered="false">
        </targetElement>
    </relation>

    <!-- Many-to-Many: Products belong to many Categories and vice versa -->
    <relation code="CategoryProductRelation" localized="false">
        <deployment table="cat2prodrel" typecode="143"/>
        <sourceElement type="Category" qualifier="supercategories" cardinality="many"
                       collectiontype="set" ordered="false">
        </sourceElement>
        <targetElement type="Product" qualifier="products" cardinality="many"
                       collectiontype="list" ordered="true">
        </targetElement>
    </relation>

    <!-- One-to-Many with partof (composition) -->
    <relation code="Order2OrderEntryRelation" localized="false">
        <sourceElement type="Order" qualifier="order" cardinality="one">
            <modifiers optional="false"/>
        </sourceElement>
        <targetElement type="OrderEntry" qualifier="entries" cardinality="many"
                       collectiontype="list" ordered="true">
            <modifiers partof="true"/>
        </targetElement>
    </relation>
</relations>
```

#### How Relations Map to the Database

**One-to-Many** relations add a foreign key column on the "many" side:
```
orders table:
┌────────┬──────────┬──────────────┐
│  PK    │  code    │  customer_pk │  ← FK to customers table
└────────┴──────────┴──────────────┘
```

**Many-to-Many** relations create a **link table** (specified in the `<deployment>`):
```
cat2prodrel table:
┌────────┬─────────────┬────────────┬──────────┐
│  PK    │  source_pk  │  target_pk │  seqnr   │
│        │ (Category)  │ (Product)  │ (order)  │
└────────┴─────────────┴────────────┴──────────┘
```

#### Key Relation Attributes

- **`qualifier`**: The attribute name on the owning type. `qualifier="orders"` on the target means `customer.getOrders()` will be available
- **`cardinality`**: `one` or `many`
- **`collectiontype`**: `list`, `set`, or `collection` — for the "many" side
- **`ordered="true"`**: Maintains insertion order (adds a sequence number column)
- **`partof="true"`**: Composition semantics — deleting the source deletes all targets. E.g., deleting an Order deletes its OrderEntries
- **`localized="true"`**: Creates a localized relation (rare, used for language-dependent associations)

#### The `partof` Semantics in Detail

`partof="true"` is often misunderstood. It means:

1. **Cascade delete**: When the parent is deleted, all children are deleted
2. **Deep clone**: When the parent is cloned (e.g., during catalog sync), children are cloned too
3. **Lifecycle coupling**: Children cannot exist without the parent

This is the difference between **aggregation** (child can exist independently) and **composition** (child's lifecycle is bound to parent). OrderEntries are `partof` Order because an OrderEntry without an Order is meaningless.

### `<itemtypes>`

This is the main section where you define item types:

```xml
<itemtypes>
    <!-- Creating a new type -->
    <itemtype code="LoyaltyTransaction" autocreate="true" generate="true"
              extends="GenericItem"
              jaloclass="com.mycompany.jalo.LoyaltyTransaction">
        <deployment table="loyalty_tx" typecode="25000"/>
        <custom-properties>
            <property name="catalogItemType">
                <value>java.lang.Boolean.FALSE</value>
            </property>
        </custom-properties>
        <attributes>
            <attribute qualifier="code" type="java.lang.String">
                <description>Unique transaction code</description>
                <modifiers unique="true" optional="false" initial="true"/>
                <persistence type="property"/>
            </attribute>
            <attribute qualifier="points" type="java.lang.Integer">
                <modifiers optional="false"/>
                <persistence type="property"/>
                <defaultvalue>Integer.valueOf(0)</defaultvalue>
            </attribute>
            <attribute qualifier="description" type="localized:java.lang.String">
                <modifiers optional="true"/>
                <persistence type="property"/>
            </attribute>
            <attribute qualifier="loyaltyScore" type="java.lang.Double">
                <modifiers read="true" write="false" optional="true"/>
                <persistence type="dynamic" attributeHandler="loyaltyScoreHandler"/>
            </attribute>
        </attributes>
        <indexes>
            <index name="codeIdx" unique="true">
                <key attribute="code"/>
            </index>
        </indexes>
    </itemtype>

    <!-- Extending an existing OOTB type -->
    <itemtype code="Customer" autocreate="false" generate="false">
        <attributes>
            <attribute qualifier="loyaltyTier" type="LoyaltyTier">
                <modifiers optional="true"/>
                <persistence type="property"/>
            </attribute>
            <attribute qualifier="loyaltyPoints" type="java.lang.Integer">
                <defaultvalue>Integer.valueOf(0)</defaultvalue>
                <modifiers optional="true"/>
                <persistence type="property"/>
            </attribute>
        </attributes>
    </itemtype>
</itemtypes>
```

#### Key `<itemtype>` Attributes

| Attribute | Description |
|-----------|-------------|
| `code` | Unique type identifier |
| `extends` | Parent type (default: `GenericItem`) |
| `autocreate` | `true` = create new type; `false` = extend existing type |
| `generate` | `true` = generate Java Model class; `false` = skip generation (used when extending) |
| `jaloclass` | Legacy Jalo layer class (required but mostly unused in ServiceLayer code) |
| `abstract` | `true` = cannot be instantiated directly |
| `singleton` | `true` = only one instance can exist |

#### The `<deployment>` Element

```xml
<deployment table="loyalty_tx" typecode="25000"/>
```

- **`table`**: The database table name for this type
- **`typecode`**: A unique integer identifying this type in the system. The platform uses typecodes internally for persistence, serialization, and the `composedtypes` table

**Typecode rules:**
- Platform types use 0–9999
- Custom types should use 10000+ (SAP recommends starting at 10000)
- Each typecode must be globally unique across all extensions
- Once assigned, a typecode **cannot be changed** without data loss
- The `ydeployments` table tracks all typecode assignments

If you don't specify a `<deployment>`, the type's attributes are stored in the parent type's table (**single-table inheritance**) or overflow into the `props` table.

## Attribute Persistence Modes

Every attribute in `items.xml` has a persistence mode that determines how its value is stored:

### `property` — Database Column

```xml
<persistence type="property"/>
```

The standard mode. Creates a column in the type's deployment table. This is what you use 90% of the time. The column type is determined by the attribute's Java type:

| Java Type | Database Column (HANA) | Database Column (MySQL) |
|-----------|----------------------|------------------------|
| `java.lang.String` | NVARCHAR(255) | VARCHAR(255) |
| `java.lang.Integer` | INTEGER | INT |
| `java.lang.Long` | BIGINT | BIGINT |
| `java.lang.Double` | DOUBLE | DOUBLE |
| `java.lang.Boolean` | TINYINT | TINYINT |
| `java.util.Date` | TIMESTAMP | DATETIME |
| `java.math.BigDecimal` | DECIMAL(30,8) | DECIMAL(30,8) |

You can customize column size with:

```xml
<persistence type="property">
    <columntype database="mysql">
        <value>VARCHAR(1000)</value>
    </columntype>
    <columntype database="hana">
        <value>NVARCHAR(1000)</value>
    </columntype>
    <columntype>
        <value>java.math.BigDecimal</value>
    </columntype>
</persistence>
```

### `dynamic` — Computed at Runtime

```xml
<persistence type="dynamic" attributeHandler="myAttributeHandler"/>
```

Dynamic attributes have **no database column**. Instead, a Spring bean (`DynamicAttributeHandler`) computes the value on every read:

```java
public class LoyaltyScoreHandler implements DynamicAttributeHandler<Double, CustomerModel> {
    
    @Override
    public Double get(CustomerModel model) {
        Integer points = model.getLoyaltyPoints();
        if (points == null) return 0.0;
        // Complex calculation based on points, tier, and activity
        double multiplier = getMultiplierForTier(model.getLoyaltyTier());
        return points * multiplier;
    }
    
    @Override
    public void set(CustomerModel model, Double value) {
        // Read-only attribute — throw exception or ignore
        throw new UnsupportedOperationException("loyaltyScore is read-only");
    }
    
    private double getMultiplierForTier(LoyaltyTier tier) {
        if (tier == null) return 1.0;
        switch (tier.getCode()) {
            case "PLATINUM": return 2.0;
            case "GOLD": return 1.5;
            case "SILVER": return 1.25;
            default: return 1.0;
        }
    }
}
```

Register in Spring:

```xml
<bean id="loyaltyScoreHandler"
      class="com.mycompany.handlers.LoyaltyScoreHandler"/>
```

**Use dynamic attributes when:**
- The value is computed from other attributes
- The value comes from an external system (API call, cache lookup)
- You want a virtual attribute without database storage overhead
- The value changes frequently and caching would be stale

**Avoid dynamic attributes when:**
- You need to query/filter by this attribute in FlexibleSearch (dynamic attributes can't be used in WHERE clauses)
- The computation is expensive and the attribute is read frequently
- You need the value to persist across restarts

### `jalo` — Legacy Mode

```xml
<persistence type="jalo"/>
```

Jalo persistence means the attribute getter/setter is implemented in the Jalo layer class. This is **deprecated** — never use this for new attributes. It exists only for backward compatibility with very old extensions.

### `cmp` — Container Managed Persistence

An internal persistence mode used by the platform for certain system attributes. You should never use `cmp` for custom attributes.

## Attribute Modifiers Deep Dive

The `<modifiers>` element controls attribute behavior:

```xml
<modifiers read="true" write="true" optional="true" unique="false"
           initial="false" search="true" encrypted="false"
           dontOptimize="false" partof="false" removable="true"/>
```

| Modifier | Default | Description |
|----------|---------|-------------|
| `read` | `true` | Whether the attribute can be read (getter exists) |
| `write` | `true` | Whether the attribute can be written (setter exists) |
| `optional` | `true` | If `false`, the attribute is mandatory — `modelService.save()` will fail without a value |
| `unique` | `false` | If `true`, the database column has a UNIQUE constraint |
| `initial` | `false` | If `true`, the value can only be set during creation — it's immutable after first save |
| `search` | `true` | If `false`, the attribute is excluded from FlexibleSearch optimization |
| `encrypted` | `false` | If `true`, the value is encrypted in the database using the platform's encryption mechanism |
| `dontOptimize` | `false` | If `true`, disables read optimization (forces DB read instead of cache) |
| `partof` | `false` | On relation attributes: defines composition (child lifecycle bound to parent) |
| `removable` | `true` | If `false`, the attribute cannot be removed from the type system |

### Practical Modifier Combinations

**Immutable identifier:**
```xml
<attribute qualifier="code" type="java.lang.String">
    <modifiers unique="true" optional="false" initial="true" write="false"/>
    <persistence type="property"/>
</attribute>
```

**Sensitive data (encrypted):**
```xml
<attribute qualifier="taxId" type="java.lang.String">
    <modifiers encrypted="true" optional="true"/>
    <persistence type="property"/>
</attribute>
```

**Read-only computed field:**
```xml
<attribute qualifier="totalValue" type="java.math.BigDecimal">
    <modifiers read="true" write="false" optional="true"/>
    <persistence type="dynamic" attributeHandler="orderTotalHandler"/>
</attribute>
```

## Type Deployment: How Types Map to Database Tables

Understanding how the type system maps to relational storage is critical for debugging and performance.

### Dedicated Table Deployment

When you specify `<deployment table="my_table" typecode="25000"/>`, the type gets its own table:

```
loyalty_tx table:
┌──────────────┬─────────────┬────────┬──────────┬─────────────────┬───────────┐
│ PK (BIGINT)  │ TypePkString│ p_code │ p_points │ p_txdate        │OwnerPkStr │
│              │             │        │          │                 │           │
│ 8796093087744│ 8796...     │ LT-001 │ 500      │ 2025-01-15 10:30│ null      │
└──────────────┴─────────────┴────────┴──────────┴─────────────────┴───────────┘
```

Key columns automatically added by the platform:

| Column | Purpose |
|--------|---------|
| `PK` | Primary key (BIGINT) — generated by the PK generator |
| `TypePkString` | The PK of the `ComposedType` record (enables polymorphic queries) |
| `p_*` | Attribute columns (prefixed with `p_` by convention) |
| `OwnerPkString` | Owner reference (used for `partof` relationships) |
| `createdTS` | Creation timestamp |
| `modifiedTS` | Last modification timestamp |
| `aCLTS` | Access control timestamp |
| `propTS` | Properties timestamp |

### Single-Table Inheritance

If a type does **not** have a `<deployment>`, its attributes are stored in the parent type's table. For example, `Customer` extends `User`, and if `Customer` has no separate deployment, its custom attributes go into the `users` table:

```
users table:
┌────────┬──────────┬──────────┬────────────────┬───────────────┐
│  PK    │  p_uid   │  p_name  │ p_loyaltytier  │ p_loyaltypts  │
│        │          │          │ (from Customer)│ (from Customer)│
└────────┴──────────┴──────────┴────────────────┴───────────────┘
```

This is efficient for small additions but can bloat the parent table if many subtypes add many attributes.

### The `props` Table (Overflow)

When an attribute cannot fit in the type's deployment table (e.g., the type has no dedicated deployment, or the attribute is added to a type after initialization without running `updatesystem`), it goes to the **`props`** table:

```
props table:
┌────────────┬───────────┬────────────────┬──────────────┐
│  ITEMPK    │  NAME     │  LANGPK        │  VALUE1      │
│            │           │                │              │
│ 8796...    │ myattr    │ null           │ some_value   │
└────────────┴───────────┴────────────────┴──────────────┘
```

The `props` table is essentially a key-value store. It's flexible but **terrible for performance** — every attribute access requires a separate lookup. This is why you should:

1. Always define a `<deployment>` for new types
2. Run `ant updatesystem` after adding attributes to existing types
3. Monitor `props` table usage with: 

```sql
SELECT COUNT(*) FROM props;
-- If this is large (millions+), investigate which attributes are overflowing
```

### The `ydeployments` Table

The platform maintains a `ydeployments` table that maps typecodes to table names:

```sql
SELECT * FROM ydeployments;
```

```
┌───────────┬─────────────────┬────────────────────┐
│ Typecode  │ TableName       │ ExtensionName      │
├───────────┼─────────────────┼────────────────────┤
│ 1         │ items           │ core               │
│ 23        │ products        │ catalog            │
│ 25000     │ loyalty_tx      │ myextension        │
└───────────┴─────────────────┴────────────────────┘
```

## Generated Model Classes

When you run `ant build` (or `ant all`), the platform reads all `items.xml` files and generates Java Model classes.

### The Generation Pipeline

```
items.xml (all extensions)
         │
         ▼
┌─────────────────────────┐
│   ItemsXML Merge        │  Combines all items.xml into unified type definitions
│   (TypeSystemCompiler)  │
└────────┬────────────────┘
         │
         ▼
┌─────────────────────────┐
│   Model Generator       │  Generates Java source files
│   (ModelGenerator)      │
└────────┬────────────────┘
         │
         ▼
┌─────────────────────────┐
│   gensrc/ directory     │  GeneratedLoyaltyTransactionModel.java
│                         │  (DO NOT EDIT)
└────────┬────────────────┘
         │
         ▼
┌─────────────────────────┐
│   src/ directory        │  LoyaltyTransactionModel.java (your class)
│                         │  extends GeneratedLoyaltyTransactionModel
└─────────────────────────┘
```

### Generated Code Structure

For a `LoyaltyTransaction` type, the generator creates:

```java
// gensrc/com/mycompany/model/GeneratedLoyaltyTransactionModel.java
// *** DO NOT EDIT THIS FILE ***

@SuppressWarnings("all")
public class GeneratedLoyaltyTransactionModel extends ItemModel {
    
    // Attribute constants
    public static final String CODE = "code";
    public static final String POINTS = "points";
    public static final String DESCRIPTION = "description";
    public static final String LOYALTYSCORE = "loyaltyScore";
    public static final String CUSTOMER = "customer";
    public static final String TRANSACTIONDATE = "transactionDate";
    
    // Getters
    @Accessor(qualifier = "code", type = Accessor.Type.GETTER)
    public String getCode() {
        return getPersistenceContext().getPropertyValue(CODE);
    }
    
    @Accessor(qualifier = "points", type = Accessor.Type.GETTER)
    public Integer getPoints() {
        return getPersistenceContext().getPropertyValue(POINTS);
    }
    
    // Localized getter
    @Accessor(qualifier = "description", type = Accessor.Type.GETTER)
    public String getDescription() {
        return getDescription(null);
    }
    
    @Accessor(qualifier = "description", type = Accessor.Type.GETTER)
    public String getDescription(Locale loc) {
        return getPersistenceContext().getLocalizedValue(DESCRIPTION, loc);
    }
    
    // Dynamic attribute getter (no setter because write="false")
    @Accessor(qualifier = "loyaltyScore", type = Accessor.Type.GETTER)
    public Double getLoyaltyScore() {
        return getPersistenceContext().getDynamicValue(this, LOYALTYSCORE);
    }
    
    // Setters
    @Accessor(qualifier = "code", type = Accessor.Type.SETTER)
    public void setCode(String value) {
        getPersistenceContext().setPropertyValue(CODE, value);
    }
    
    @Accessor(qualifier = "points", type = Accessor.Type.SETTER)
    public void setPoints(Integer value) {
        getPersistenceContext().setPropertyValue(POINTS, value);
    }
    
    // Localized setter
    @Accessor(qualifier = "description", type = Accessor.Type.SETTER)
    public void setDescription(String value) {
        setDescription(value, null);
    }
    
    @Accessor(qualifier = "description", type = Accessor.Type.SETTER)
    public void setDescription(String value, Locale loc) {
        getPersistenceContext().setLocalizedValue(DESCRIPTION, loc, value);
    }
}
```

### PersistenceContext Internals

The `PersistenceContext` is the engine behind Model classes:

- **Dirty tracking**: When you call `setPoints(100)`, the PersistenceContext marks the attribute as dirty. Only dirty attributes are written to the database on `modelService.save()`.
- **Lazy loading**: Relation attributes (e.g., `getCustomer()`) are loaded lazily — the database query only happens when you first access the attribute.
- **Cache integration**: Values are cached in the PersistenceContext. Calling `getCode()` twice doesn't hit the database twice.
- **Dynamic dispatch**: For `type="dynamic"` attributes, the PersistenceContext delegates to the registered `DynamicAttributeHandler` Spring bean.

### Your Model Class

You create a non-generated class that extends the generated one:

```java
// src/com/mycompany/model/LoyaltyTransactionModel.java
public class LoyaltyTransactionModel extends GeneratedLoyaltyTransactionModel {
    // Add custom convenience methods here
    
    public boolean isHighValue() {
        return getPoints() != null && getPoints() > 1000;
    }
}
```

**Never edit files in `gensrc/`** — they're overwritten on every build.

## The Runtime Type System

The type system isn't just a build-time concept — it's fully inspectable at runtime.

### TypeManager

```java
TypeManager typeManager = TypeManager.getInstance();

// Get a ComposedType
ComposedType productType = typeManager.getComposedType("Product");

// Get all attributes including inherited ones
Set<AttributeDescriptor> allAttributes = productType.getAttributeDescriptors();

// Get only declared attributes (not inherited)
Set<AttributeDescriptor> declaredAttributes = productType.getDeclaredAttributeDescriptors();

// Check type hierarchy
ComposedType superType = productType.getSuperType();  // GenericItem or parent
Collection<ComposedType> subTypes = productType.getSubTypes();

// Check if a type is assignable to another
boolean isProduct = typeManager.isAssignableTo(someType, productType);
```

### AttributeDescriptor

```java
AttributeDescriptor codeAttr = productType.getAttributeDescriptor("code");

// Inspect attribute metadata
String qualifier = codeAttr.getQualifier();      // "code"
Type attrType = codeAttr.getAttributeType();      // AtomicType for java.lang.String
boolean isOptional = codeAttr.isOptional();        // false
boolean isUnique = codeAttr.isUnique();            // true
boolean isLocalized = codeAttr.isLocalized();      // false
boolean isWritable = codeAttr.isWritable();        // true
String persistenceType = codeAttr.getPersistenceClass(); // property, dynamic, etc.
```

### Programmatic Type Queries

This is useful for building dynamic tools, backoffice extensions, or migration scripts:

// List all custom types (typecode >= 10000)
TypeManager tm = TypeManager.getInstance();

        }
    }
}

## Type System and FlexibleSearch

FlexibleSearch is the query language for the type system. Understanding how type system concepts map to queries is essential.

### Curly Brace Syntax Resolution

When you write:
```sql
SELECT {pk}, {code}, {name} FROM {Product}
```

The platform resolves:
- `{Product}` → looks up the `ComposedType` for "Product", finds its deployment table (`products`)
- `{pk}` → maps to the `PK` column
- `{code}` → finds the `AttributeDescriptor` for "code", maps to `p_code` column
- `{name}` → finds it's localized, joins the `productslp` table filtered by session language

The generated SQL (simplified) is:
```sql
SELECT item.PK, item.p_code, lp.p_name 
FROM products item 
LEFT JOIN productslp lp ON item.PK = lp.ITEMPK AND lp.LANGPK = ?sessionLanguagePK
```

### Polymorphic Queries

Because the type system tracks inheritance, FlexibleSearch supports polymorphic queries:

```sql
-- Returns Products AND all subtypes (like ApparelProduct, ElectronicsProduct)
SELECT {pk} FROM {Product}

-- Returns ONLY exact Product instances (no subtypes)
SELECT {pk} FROM {Product!}

-- The ! suffix restricts to the exact type
```

### Type-Aware Joins

```sql
-- Join using relation definitions
SELECT {o.pk}, {o.code}, {c.uid}
FROM {Order AS o JOIN Customer AS c ON {o.customer} = {c.pk}}
WHERE {c.loyaltyTier} = ?tier

-- The platform resolves {o.customer} to the FK column automatically
```

### Querying Enums

```sql
-- Enum values are referenced by their enumeration code
SELECT {pk} FROM {Customer} WHERE {loyaltyTier} = 
    ({{SELECT {pk} FROM {EnumerationValue} WHERE {code} = 'GOLD' 
       AND {Type} = ({{SELECT {pk} FROM {ComposedType} WHERE {code} = 'LoyaltyTier'}})}})

-- Or using the simpler HybrisEnumValue approach with parameters
FlexibleSearchQuery query = new FlexibleSearchQuery(
    "SELECT {pk} FROM {Customer} WHERE {loyaltyTier} = ?tier");
query.addQueryParameter("tier", LoyaltyTier.valueOf("GOLD"));
```

## Type System Evolution: Schema Migration

When you modify `items.xml` (add types, add attributes, change modifiers), you need to update the database schema.

### `ant updatesystem`

This is the safe way to evolve the schema. It:

1. Reads the current database schema
2. Reads the merged type system from all `items.xml` files
3. Computes the diff
4. Generates and executes DDL statements (ALTER TABLE ADD COLUMN, CREATE TABLE, CREATE INDEX, etc.)
5. **Never drops columns or tables** — only additive changes

```bash
ant updatesystem
```

### `ant initialize`

This **destroys all data** and recreates the schema from scratch:

1. Drops all tables
2. Recreates the full schema
3. Loads essential data (from `essentialdata-*.impex`)
4. Loads project data (from `projectdata-*.impex`)

**Never run `ant initialize` on production.**

### What You Can and Cannot Change

| Change | `updatesystem` Safe? | Notes |
|--------|---------------------|-------|
| Add new type | ✅ | Creates new table |
| Add attribute to existing type | ✅ | ALTER TABLE ADD COLUMN |
| Add index | ✅ | CREATE INDEX |
| Change attribute type | ⚠️ | May fail if types are incompatible |
| Remove attribute | ❌ | Column remains in DB (not dropped) |
| Remove type | ❌ | Table remains in DB (not dropped) |
| Change typecode | ❌ | **Data corruption** — never do this |
| Change deployment table | ❌ | Existing data becomes orphaned |

### The `SchemaGenerator` Internal

The `TypeSystemSetup` class orchestrates schema migration. The `DatabaseSchemaGenerator` compares the expected schema (from type system metadata) with the actual database schema (from JDBC `DatabaseMetaData`) and generates DDL. The internal process:

```
TypeSystem metadata (items.xml)  ←→  Database metadata (JDBC)
              │                              │
              └──────────┬───────────────────┘
                         │
                    ┌────▼─────┐
                    │  Diff    │
                    │  Engine  │
                    └────┬─────┘
                         │
                    ┌────▼──────────────┐
                    │  DDL Statements    │
                    │  ALTER TABLE ...   │
                    │  CREATE TABLE ...  │
                    │  CREATE INDEX ...  │
                    └───────────────────┘
```

## Indexes

Indexes in `items.xml` map directly to database indexes:

```xml
<indexes>
    <!-- Simple index -->
    <index name="codeIdx">
        <key attribute="code"/>
    </index>

    <!-- Unique index (also creates a unique constraint) -->
    <index name="uniqueCodeIdx" unique="true">
        <key attribute="code"/>
    </index>

    <!-- Composite index -->
    <index name="customerDateIdx">
        <key attribute="customer"/>
        <key attribute="transactionDate"/>
    </index>
</indexes>
```

Generated DDL:
```sql
CREATE INDEX codeIdx ON loyalty_tx (p_code);
CREATE UNIQUE INDEX uniqueCodeIdx ON loyalty_tx (p_code);
CREATE INDEX customerDateIdx ON loyalty_tx (p_customer, p_txdate);
```

### Index Best Practices

1. **Index attributes used in WHERE clauses** of your FlexibleSearch queries
2. **Composite indexes** should follow the leftmost prefix rule — put the most selective column first
3. **Don't over-index** — each index slows down writes
4. **Unique indexes** on `code` or `uid` attributes are almost always a good idea
5. **Monitor slow queries** in HAC and add indexes as needed

## Best Practices

### Naming Conventions

| Element | Convention | Example |
|---------|-----------|---------|
| Type codes | PascalCase, descriptive | `LoyaltyTransaction`, `CustomProduct` |
| Attribute qualifiers | camelCase | `loyaltyPoints`, `transactionDate` |
| Relation codes | `Source2TargetRelation` | `Customer2LoyaltyTransactionRelation` |
| Enum codes | PascalCase | `LoyaltyTier`, `TransactionType` |
| Table names | lowercase, underscore | `loyalty_tx`, `custom_products` |
| Index names | camelCase + `Idx` suffix | `codeIdx`, `customerDateIdx` |

### Typecode Management

- Maintain a typecode registry document in your project
- Assign ranges per extension (e.g., `mycore`: 25000–25099, `myfacades`: 25100–25199)
- Never reuse a typecode that was previously assigned to a different type
- In multi-team projects, agree on typecode ranges to avoid collisions

### When to Create New Types vs. Extend Existing

**Create a new type when:**
- The entity represents a new domain concept (e.g., `LoyaltyTransaction`)
- You need dedicated table storage for performance
- The entity has its own lifecycle (CRUD operations)

**Extend an existing type when:**
- You're adding attributes to OOTB types (e.g., adding `loyaltyTier` to `Customer`)
- The additional data logically belongs to the existing type
- You don't need a separate table

### Performance Considerations

1. **Keep type tables focused**: Don't add 50 attributes to a single type — consider splitting into related types
2. **Use dedicated deployments**: Types without deployments use single-table inheritance or the `props` table — both have performance costs at scale
3. **Watch collection-type attributes**: They're serialized into a single column — terrible for large collections. Use relations instead
4. **Localized attributes add JOIN overhead**: Every localized attribute requires a JOIN to the `*lp` table. Only localize what truly needs multiple languages
5. **Dynamic attributes are computed per access**: Cache-intensive code shouldn't call dynamic attribute getters in tight loops

## Common Mistakes and How to Avoid Them

### 1. Changing a Typecode After Deployment

**Problem:** Changing a typecode breaks the mapping between existing database rows and the type system. All data for that type becomes inaccessible.

**Solution:** Never change typecodes. If you need to restructure, create a new type with a new typecode and migrate data.

### 2. Using Collection Types for Large Lists

**Problem:**
```xml
<!-- DON'T DO THIS for large collections -->
<attribute qualifier="tags" type="StringCollection">
    <persistence type="property"/>
</attribute>
```
Collection-type attributes serialize the entire collection into one column. With thousands of items, this causes performance issues and potential data truncation.

**Solution:** Use a relation instead:
```xml
<relation code="Product2TagRelation">
    <sourceElement type="Product" qualifier="product" cardinality="one"/>
    <targetElement type="Tag" qualifier="tags" cardinality="many"/>
</relation>
```

### 3. Forgetting `autocreate="false"` When Extending

**Problem:**
```xml
<!-- WRONG: This tries to CREATE a new Customer type -->
<itemtype code="Customer" autocreate="true" generate="true">
```

**Solution:**
```xml
<!-- RIGHT: This EXTENDS the existing Customer type -->
<itemtype code="Customer" autocreate="false" generate="false">
```

### 4. Not Running `updatesystem` After Changes

**Problem:** After adding attributes to `items.xml` and building, the new attributes go to the `props` table instead of proper columns because the DB schema wasn't updated.

**Solution:** Always run `ant updatesystem` after changing `items.xml` (or redeploy in CCv2, which runs updatesystem automatically).

### 5. Querying Dynamic Attributes in FlexibleSearch

**Problem:**
```sql
-- This FAILS — dynamic attributes have no database column
SELECT {pk} FROM {Customer} WHERE {loyaltyScore} > 100
```

**Solution:** If you need to query by a computed value, either:
- Store it as a `property` attribute and update it via an interceptor or cron job
- Compute the filter in Java after fetching results
- Use a different query strategy (e.g., query by the underlying attributes)

### 6. Ignoring `partof` Semantics

**Problem:** Not using `partof="true"` for composition relationships means child objects aren't deleted when the parent is deleted, leading to orphaned data.

**Solution:** Use `partof="true"` for entities that logically belong to their parent (OrderEntry → Order, AddressEntry → Address).

### 7. Modifying OOTB items.xml Files

**Problem:** Editing items.xml files inside `bin/platform/` or `bin/modules/` will be overwritten on upgrade.

**Solution:** Always extend types from your own extension's items.xml using `autocreate="false" generate="false"`.

## Summary

The SAP Commerce type system is fundamentally different from traditional ORM approaches, and understanding it deeply is essential for effective Commerce development:

1. **Metadata-driven**: XML definitions in `items.xml` are the single source of truth for data modeling — the platform generates Java code and database schemas from them
2. **Extensible by design**: Any extension can add attributes to any type using `autocreate="false"`, and the platform merges all definitions at build time
3. **Multiple persistence modes**: `property` (database column), `dynamic` (computed at runtime), and legacy `jalo` — choose based on your use case
4. **Rich relationship model**: One-to-many, many-to-many, ordered collections, and `partof` composition semantics are all first-class concepts
5. **Built-in localization**: The `localized:` type prefix and `*lp` tables provide multi-language support without any custom implementation
6. **Runtime introspection**: `TypeManager`, `ComposedType`, and `AttributeDescriptor` let you inspect the entire type system programmatically
7. **Automatic schema evolution**: `ant updatesystem` handles DDL generation — you never write `ALTER TABLE` statements manually
8. **FlexibleSearch integration**: The query language understands types, inheritance, and localization natively

Master the type system, and you master the foundation that every other SAP Commerce feature is built upon.
