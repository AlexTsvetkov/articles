## Overview

This article provides a comprehensive walkthrough of SAP Commerce Cloud's internal architecture for developers new to the platform or seeking deeper understanding of its internals. It covers the extension system, type system, service layer architecture, core design patterns, and customization best practices.

**Target audience:** Java developers working with or evaluating SAP Commerce Cloud.

**Applies to:** SAP Commerce Cloud 2211 and later (concepts apply to earlier versions as well).

---

## Problem Statement

SAP Commerce Cloud has a steep learning curve. Its architecture differs significantly from standard Spring Boot applications — the type system replaces JPA/Hibernate, extensions replace modules, and a unique set of design patterns (Populators, Strategies, Interceptors) govern how customizations are implemented.

Without a clear understanding of these architectural foundations, development teams risk making decisions that lead to upgrade difficulties, performance issues, and unmaintainable codebases.

---

## Platform Overview

SAP Commerce Cloud (formerly SAP Hybris Commerce) is a Java-based enterprise e-commerce platform within the SAP Customer Experience (CX) portfolio. It runs as a Spring application on embedded Tomcat.

Key characteristics:
- **Modular monolith** — all features are packaged as extensions within a single deployable unit
- **Dynamic type system** — data models defined in XML (`items.xml`), with Java classes and database schemas generated by the platform
- **OCC REST APIs** — headless commerce APIs consumed by SAP Composable Storefront (Spartacus) or custom frontends
- **SAP Commerce Cloud v2 (CCv2)** — managed cloud deployment on Microsoft Azure

> **Reference:** [SAP Commerce Cloud Architecture Overview — SAP Help Portal](https://help.sap.com/docs/SAP_COMMERCE_CLOUD)

## The Extension System

Everything in SAP Commerce Cloud is an extension — platform core, commerce modules, and custom project code.

### Extension Structure

```
myextension/
├── extensioninfo.xml          # Extension metadata and dependencies
├── resources/
│   ├── myextension-items.xml  # Type System definitions
│   └── myextension-spring.xml # Spring bean definitions
├── src/                       # Java source code
├── gensrc/                    # Generated model classes
└── web/                       # Web module (controllers, etc.)
```

Dependencies are declared in `extensioninfo.xml` via `<requires-extension>` elements. The platform resolves the complete dependency graph at startup and loads extensions in order.

### Extension Categories

| Category | Location | Examples |
|---|---|---|
| Platform | `bin/platform/ext/` | `core`, `processing`, `Europe1` |
| Commerce Modules | `bin/modules/` | `commerceservices`, `cms2`, `b2bcommerce` |
| Custom | `bin/custom/` | Project-specific extensions |

### CCv2 Configuration

In SAP Commerce Cloud v2, the extension set is controlled via `manifest.json`:

```json
{
  "extensions": [
    "commercewebservices",
    "yacceleratorstorefront",
    "mycustomextension"
  ]
}
```

The build system generates `localextensions.xml` from this configuration.

> **Reference:** [SAP Commerce Cloud Build Manifest — SAP Help Portal](https://help.sap.com/docs/SAP_COMMERCE_CLOUD)

## The Type System

The type system is the most distinctive feature of SAP Commerce Cloud. Instead of JPA `@Entity` annotations, data models are defined in XML.

### Defining a Custom Type

```xml
<itemtype code="LoyaltyTransaction" autocreate="true" generate="true"
          extends="GenericItem">
    <deployment table="loyalty_transactions" typecode="25000"/>
    <attributes>
        <attribute qualifier="code" type="java.lang.String">
            <modifiers unique="true" optional="false"/>
            <persistence type="property"/>
        </attribute>
        <attribute qualifier="points" type="java.lang.Integer">
            <modifiers optional="false"/>
            <persistence type="property"/>
        </attribute>
        <attribute qualifier="loyaltyScore" type="java.lang.Double">
            <persistence type="dynamic" 
                         attributeHandler="loyaltyScoreHandler"/>
        </attribute>
    </attributes>
</itemtype>
```

**Key concepts:**

| Element | Purpose |
|---|---|
| `deployment` | Maps type to a database table with unique typecode |
| `persistence type="property"` | Standard database column |
| `persistence type="dynamic"` | Computed at runtime by a Spring bean |
| `autocreate="false" generate="false"` | Extends an existing type from another extension |

### Comparison: Type System vs. JPA

| JPA/Hibernate | SAP Commerce Cloud |
|---|---|
| `@Entity` annotations | `items.xml` definitions |
| Flyway/Liquibase migrations | `ant updatesystem` / System Update in HAC |
| Developer writes entity classes | Platform generates model classes |
| JPQL queries | FlexibleSearch queries |

> **Reference:** [Type System — SAP Help Portal](https://help.sap.com/docs/SAP_COMMERCE/d0224eca81e249cb821f2cdf45a82ace/8bc399c186691014a895f3ced4e6f8bb.html)

## The Service Layer Architecture

SAP Commerce Cloud enforces a strict layered architecture:

**Controller → Facade → Service → DAO → ModelService**

| Layer | Responsibility |
|---|---|
| **Controller** | HTTP request/response handling, delegates to Facades |
| **Facade** | Orchestration, Model-to-DTO conversion, transaction boundary |
| **Service** | Core business logic, operates on Model objects |
| **DAO** | Data access via FlexibleSearch |
| **ModelService** | Persistence operations (`create()`, `save()`, `remove()`) |

### Important Architecture Rules

- Business logic must reside in Services, not Controllers
- FlexibleSearch queries belong in DAOs, not Services
- Model-to-DTO conversion uses the Populator/Converter pattern
- Facades define the transaction boundary

> **Reference:** [ServiceLayer Architecture — SAP Help Portal](https://help.sap.com/docs/SAP_COMMERCE/d0224eca81e249cb821f2cdf45a82ace/8c6b2c5e86691014b2e1b37daab54a2d.html)

## Core Design Patterns

### Populator/Converter Pattern

Converts Model objects to Data Transfer Objects (DTOs) via composable populators:

```xml
<bean id="productConverter" parent="abstractPopulatingConverter">
    <property name="populators">
        <list>
            <ref bean="productBasicPopulator"/>
            <ref bean="productPricePopulator"/>
        </list>
    </property>
</bean>
```

Custom extensions can add populators using `<list merge="true">` — enabling extension without modification of existing code.

### Strategy Pattern

Business decisions are implemented as pluggable strategies (e.g., `FindPriceStrategy`, `CommerceAddToCartStrategy`). Override the Spring bean to change behavior.

### Interceptor Chain

Lifecycle hooks on ModelService operations:

| Interceptor | Triggered |
|---|---|
| `InitDefaultsInterceptor` | During `modelService.create()` |
| `PrepareInterceptor` | Before save — data transformation |
| `ValidateInterceptor` | Before save — validation rules |
| `RemoveInterceptor` | Before delete |

## Customization Best Practices

| Approach | When to Use | Upgrade Risk |
|---|---|---|
| Out of the Box (OOTB) | Requirements match 80%+ | Low |
| Configuration (properties/ImpEx) | Adjustable via configuration | Low |
| Extend (add populator/strategy/interceptor) | Need additive behavior | Medium |
| Override (replace Spring bean) | Need to change existing behavior | Medium-High |
| Custom Extension | New business domain | High |
| Modify OOTB Code | **Never** | **Critical** |

> **Important:** Never modify code in `bin/platform/` or `bin/modules/`. Always implement customizations through your own extensions in `bin/custom/`.
